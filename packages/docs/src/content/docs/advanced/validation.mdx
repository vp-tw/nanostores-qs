---
title: Validation
description: Validate query parameters with Zod and other schema libraries
---

Validate via `decode` and fall back to `defaultValue` on failure. Any schema library works — Zod, Valibot, ArkType, etc.

## Zod example

```tsx
import { z } from "zod";

const SortOptionSchema = z.enum(["newest", "price_asc", "price_desc"]);
type SortOption = z.infer<typeof SortOptionSchema>;

const sort = qsUtils.createSearchParamStore("sort", {
  decode: (v) => {
    const result = SortOptionSchema.safeParse(v);
    return result.success ? result.data : "newest";
  },
  defaultValue: "newest" as SortOption,
});
```

## Custom preset with Zod

Combine `createPreset` with Zod for reusable validated presets:

```ts
import { createPreset } from "@vp-tw/nanostores-qs/presets";
import { z } from "zod";

function zodEnum<const T extends readonly [string, ...Array<string>]>(values: T) {
  const schema = z.enum(values);
  return createPreset<T[number], T[0]>({
    decode: (v) => schema.parse(String(v)),
    defaultValue: values[0],
  });
}

// Usage — call with options like built-in presets
const statusPreset = zodEnum(["active", "inactive", "pending"]);
const statusStore = qsUtils.createSearchParamStore("status", statusPreset());
const optionalStatus = qsUtils.createSearchParamStore("status", statusPreset({ optional: true }));
```

## Validation strategy

The library's approach is **decode-time validation**: invalid URL values are silently replaced with the `defaultValue`. This ensures the app always has valid state, even when users manually edit URLs.

| Approach          | Behavior                    |
| ----------------- | --------------------------- |
| Parse succeeds    | Use parsed value            |
| Parse fails       | Fall back to `defaultValue` |
| Parameter missing | Use `defaultValue`          |
